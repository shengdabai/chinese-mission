## 0. 文档摘要

### 0.1 一句话定义
不是做“中文版多邻国”，而是做一套让外国人**在真实中文场景里完成任务、不断犯错、不断修正、最后真的能说出来**的 AI 产品。

### 0.2 核心判断
根据你给出的两个参考方向——[1000h.org](https://1000h.org/) 和 [everyone-can-use-english](https://github.com/ZuodaoTech/everyone-can-use-english/)——再结合语言能力框架 [ACTFL](https://www.actfl.org/educator-resources/actfl-proficiency-guidelines/) 与 [CEFR](https://www.coe.int/en/web/common-european-framework-reference-languages)，真正有价值的路径不是：

- 背词
- 做题
- 打卡
- 记短语

而是：

- **在真实意图下说中文**
- **理解中文句子结构**
- **在误解与修正中继续对话**
- **把“学会”定义为“完成任务”**

# 1. 背景 → 现状 → 机会

## 1.1 背景

语言能力框架（如 [ACTFL](https://www.actfl.org/educator-resources/actfl-proficiency-guidelines/) 与 [CEFR](https://www.coe.int/en/web/common-european-framework-reference-languages)）强调的不是“记住多少词”，而是**在真实语境中能做什么（can-do）**。

这和你提出的判断完全一致：

> 用户不是要“学几个中文词”，而是要“在现实中把中文用出来”。

---

## 1.2 当前主流学习 App 的问题

| 问题类型 | 常见做法 | 用户表面感受 | 实际问题 |
|---|---|---|---|
| 碎片化输入 | 每天几个词、几道题 | 有完成感 | 不能迁移到真实表达 |
| 伪互动 | 选择题、排序题 | 容易上手 | 没有真实沟通压力 |
| 假掌握 | 记短语、背模板 | 觉得“学会了” | 一到现实就不会变体 |
| 过度纠错 | 每句都纠语法 | 感觉专业 | 打断表达欲望 |
| 游戏化错位 | 连续打卡、积分 | 容易上瘾 | 上瘾的是机制，不是能力 |

---

## 1.3 市场机会

### 真正的机会不是“再做一个中文学习 App”
而是做：

> **AI 驱动的中文任务模拟器 / 中文生存世界 / 中文表达训练器**

### 为什么现在机会成熟
| 条件 | 以前难点 | 现在可行原因 |
|---|---|---|
| 对话生成 | 场景内容难以规模化 | LLM 可生成多轮对话与变化 |
| 个性化反馈 | 人工老师成本高 | AI 可实时复盘与改写 |
| 结构直译 | 逐句制作成本大 | 可用规则 + LLM + 人工审核 |
| 语音练习 | 语音交互重 | ASR/TTS 已足够做 MVP |
| 学习画像 | 难持续跟踪 | 现在可自动记录错误模式 |

# 2. 参考产品拆解

> 说明：以下分析以你提供的参考链接公开定位和方法导向为基础，重点提炼**产品哲学**，不是逐像素逆向。

## 2.1 参考方法论对比

| 维度 | [1000h.org](https://1000h.org/) | [everyone-can-use-english](https://github.com/ZuodaoTech/everyone-can-use-english/) | 对本产品的启发 |
|---|---|---|---|
| 反对什么 | 碎片化、低密度、假努力 | 传统背词、做题、应试英语 | 中文产品也不能走“刷题式假学习” |
| 强调什么 | 长时间沉浸、真实接触语言 | “使用语言”而不是“学习知识” | 学习主单位应是**任务/场景** |
| 用户心智 | 习得导向 | 使用导向 | 我们要做“会用中文” |
| 方法短板 | 可能偏输入 | 方法论强但未必完整产品化 | 你的机会在于“AI 产品化” |
| 可借鉴点 | 用时间/体验替代题目数量 | 用行动替代记忆 | 用“完成任务数”替代“背词数” |

---

## 2.2 对标结论

### 应该继承的
| 应继承 | 原因 |
|---|---|
| 反“假学习” | 这是最大差异化 |
| 重真实语境 | 语言能力来自使用 |
| 强调沉浸 | 比课程感更强 |
| 让用户直接开口 | 输出闭环不可少 |

### 不应照搬的
| 不应照搬 | 原因 |
|---|---|
| 纯自然习得、过少支架 | 初学者容易迷失 |
| 只有理念、没有产品机制 | 不足以做留存和增长 |
| 只有输入、缺少任务判定 | 用户不知道自己是否“会用了” |

# 3. 产品愿景与定位

## 3.1 产品愿景
帮助外国人**在真实中文世界中活下来、聊起来、用起来**。

## 3.2 产品定位
一款基于 AI 的中文任务学习软件，围绕：

- **真实任务**
- **结构直译**
- **沉浸式对话**
- **容错练习**
- **个性化复盘**

来帮助用户建立**可迁移的中文表达能力**。

---

## 3.3 核心价值主张

| 维度 | 定义 |
|---|---|
| 不是学知识 | 是学“如何把事情说清楚” |
| 不是记词 | 是记“表达模式” |
| 不是刷题 | 是完成“中文任务” |
| 不是追求翻译优雅 | 是理解中文结构 |
| 不是怕出错 | 是在犯错中继续沟通 |

# 4. 目标用户与 JTBD

## 4.1 核心用户画像

| 用户类型 | 特征 | 痛点 | 最强需求 |
|---|---|---|---|
| 初到中国的外国人 | 生存需求强 | 会看不会说、会背不会用 | 点餐、打车、付款、问路 |
| 中文初学者 | 有兴趣但挫败高 | 学了很多、开不了口 | 真实表达、低压力练习 |
| 外企员工 | 有商务场景 | 课堂太慢、现实太快 | 自我介绍、寒暄、简单工作沟通 |
| 海外高校学生 | 有课程但缺实践 | 课内会，课外不会 | 把书本中文变成可用中文 |
| 外籍伴侣/家庭用户 | 关系场景强 | 家庭交流、见家长压力大 | 社交表达、日常礼貌沟通 |

---

## 4.2 JTBD（Jobs To Be Done）

| 场景 | Job Statement |
|---|---|
| 餐厅点单 | 当我在中国餐厅点餐时，我想用中文说出我想吃什么、不要什么、怎么付钱，即使说得不标准也能完成订单。 |
| 打车 | 当我坐出租车时，我想清楚表达目的地和路线偏好，即使司机说得快，我也能请求重复或放慢。 |
| 自我介绍 | 当别人问我是谁、来自哪里、做什么时，我想自然地用中文说出来，不靠死记硬背。 |
| 社交聊天 | 当我认识新朋友时，我想继续话题，而不是只会你好谢谢。 |
| 工作协作 | 当我在会议前后寒暄或做简单沟通时，我想说得体、被理解、不尴尬。 |

# 5. 产品原则

## 5.1 产品设计原则

| 原则 | 解释 |
|---|---|
| 任务优先 | 学习目标是完成事情，不是完成练习 |
| 先表达后纠错 | 保护交流欲，不把用户打断 |
| 结构直译优先于自然翻译 | 帮用户理解中文是怎么长出来的 |
| 有支架但不喂答案 | 让用户真正参与构句 |
| 反馈只盯高价值错误 | 只纠影响理解和高频迁移的错误 |
| 场景强于词表 | 以任务树而不是单词书组织内容 |
| AI 受控而非放飞 | 不做无限自由聊天，要围绕目标推进 |

# 6. 核心学习模型

## 6.1 Mission Loop（核心闭环）

```text
┌────────────┐
│ 选择真实任务 │
└─────┬──────┘
      ↓
┌──────────────────┐
│ 进入场景：角色/地点/目标 │
└─────┬────────────┘
      ↓
┌──────────────────┐
│ 热身：关键表达 + 结构直译 │
└─────┬────────────┘
      ↓
┌──────────────────┐
│ AI 多轮对话：边说边试错   │
└─────┬────────────┘
      ↓
┌──────────────────┐
│ 任务判定：成功/失败/部分完成 │
└─────┬────────────┘
      ↓
┌──────────────────┐
│ 复盘：修正/替换/重说/迁移   │
└─────┬────────────┘
      ↓
┌──────────────────┐
│ 进入下一相似场景或现实实践   │
└──────────────────┘
```

---

## 6.2 三层语言显示模型

这是产品的核心壁垒之一。

| 层级 | 作用 | 示例 |
|---|---|---|
| L1 中文原句 | 目标语言本体 | 你叫什么名字 |
| L2 结构直译 Gloss | 告诉用户中文结构 | you / are-called / what / name |
| L3 自然翻译 | 保证理解 | what is your name |

### 关键原则
你强调“更好的英文翻译不是重点，而是每个中文对应英文结构”，这个方向是对的。  
但产品上建议做成：

> **逐词 + 词块 + 结构混合对齐**  
而不是机械逐字一一对应。

---

## 6.3 显示策略

| 用户水平 | 默认显示 |
|---|---|
| A0-A1 | 中文 + 拼音 + 结构直译 + 自然翻译 |
| A1-A2 | 中文 + 结构直译（折叠）+ 自然翻译（按需） |
| A2-B1 | 中文优先，结构直译点击展开 |
| B1+ | 中文默认，英文支架最小化 |

# 7. 目标与非目标

## 7.1 产品目标（V1）
| 目标 | 说明 |
|---|---|
| 让用户 5 分钟内进入第一次真实对话 | 缩短首次可用时间 |
| 让用户完成 20 个高频中文任务 | 不求大而全，先求高频真需求 |
| 建立“表达库”而不是“词汇本” | 沉淀个人可用语言 |
| 形成可迁移表达模式 | 会替换、会组合、会应变 |
| 验证任务式学习比题目式学习更有效 | 通过任务完成率和留存验证 |

---

## 7.2 非目标（V1 不做）
| 不做什么 | 原因 |
|---|---|
| 大而全课程体系 | 先打透高频任务 |
| HSK 为主线 | 会把产品拉回考试逻辑 |
| 真人社交社区 | 初期复杂度高 |
| 全自动无限 AI 自由聊天 | 可控性差、教学价值不稳定 |
| 重度语法课程 | 会稀释任务体验 |
| 追求完美发音评测 | 首版先让用户敢说、会说 |

# 8. MVP 范围

## 8.1 MVP 用户范围
**英语母语/英语可用用户，中文 A0-A2 初学者**

> 原因：结构直译层先用英语桥接最合理，降低复杂度。

---

## 8.2 MVP 内容范围
| 类别 | 场景数 | 任务数 |
|---|---:|---:|
| 生存中文 | 3 | 10 |
| 社交中文 | 2 | 6 |
| 基础工作中文 | 1 | 4 |
| **总计** | **6** | **20** |

---

## 8.3 MVP 首批场景
| 场景 | 任务示例 |
|---|---|
| 咖啡店/餐厅 | 点餐、去冰、少糖、打包、付款 |
| 打车/出行 | 去哪里、走高速吗、请停这里、太快了 |
| 自我介绍 | 名字、国家、职业、来中国多久 |
| 购物 | 价格、尺寸、颜色、能不能便宜一点 |
| 问路 | 地点、方向、远不远、怎么去 |
| 听不懂求助 | 请再说一遍、慢一点、什么意思 |

## 9.2 功能模块详述

---

### 模块 A：Onboarding（首次进入）

#### 目标
快速把用户导入“真实需求”而不是课程目录。

#### 功能需求
| 编号 | 功能 | 说明 |
|---|---|---|
| A1 | 目标选择 | 生存 / 社交 / 工作 / 旅行 |
| A2 | 中文水平自评 | A0/A1/A2 或简单问卷 |
| A3 | 使用场景偏好 | 餐厅/打车/朋友/公司 |
| A4 | 学习模式选择 | 文本优先 / 语音优先 / 混合 |
| A5 | 母语或桥接语言 | MVP 先支持英文 |

#### 验收标准
| 条件 | 标准 |
|---|---|
| 首次完成时长 | ≤ 90 秒 |
| 进入首个任务时间 | ≤ 2 分钟 |
| 用户不需要先学课程 | 必须 |

---

### 模块 B：场景地图

#### 目标
让用户直接看到“我今天要完成什么”。

#### 功能需求
| 编号 | 功能 | 说明 |
|---|---|---|
| B1 | 场景分类 | 生存/社交/工作 |
| B2 | 今日推荐任务 | 根据画像推荐 |
| B3 | 任务难度标签 | A0/A1/A2 |
| B4 | 任务预计时长 | 5-10 分钟 |
| B5 | 任务成就反馈 | 完成数/连续成功率 |

---

### 模块 C：任务热身页

#### 目标
给足支架，但不把答案全喂给用户。

#### 功能需求
| 编号 | 功能 | 说明 |
|---|---|---|
| C1 | 任务目标说明 | 例：在咖啡店点一杯热拿铁 |
| C2 | 角色说明 | 店员、司机、朋友等 |
| C3 | 关键表达预览 | 2-4 个核心句型 |
| C4 | 结构直译展示 | 可展开查看 |
| C5 | 提示层级说明 | 遇卡顿可点提示 |

#### 示例
| 中文 | 结构直译 | 自然翻译 |
|---|---|---|
| 我要一杯热拿铁 | I want one cup hot latte | I’d like a hot latte |
| 不要冰的 | do not want ice one | no ice |
| 可以打包吗 | can take-away Q | can I get it to go |

---

### 模块 D：AI 对话引擎

#### 目标
模拟真实沟通，而不是老师问答题。

#### 功能需求
| 编号 | 功能 | 说明 |
|---|---|---|
| D1 | 文本输入 | 用户可打字 |
| D2 | 语音输入 | 用户可说中文 |
| D3 | NPC 语音/文本回复 | 更有沉浸感 |
| D4 | 容错理解 | 允许不完整表达 |
| D5 | 任务推进状态机 | 保证围绕任务前进 |
| D6 | 提示系统 | 结构提示 / 半句提示 / 完整参考 |
| D7 | 不打断式纠错 | 重要错误延迟反馈 |

#### 输入策略
| 模式 | 说明 |
|---|---|
| 自由说 | 用户自己表达 |
| 半引导 | 给关键词提示 |
| 填槽式支持 | 对初学者给 slot 支架 |
| 紧急求助 | 显示完整参考句 |

---

### 模块 E：任务判定

#### 目标
判断用户是否真的“完成任务”。

#### 判定逻辑
| 层级 | 定义 |
|---|---|
| 成功 | 目标已表达清楚，对方理解并任务完成 |
| 部分成功 | 目标表达不完整但核心达成 |
| 失败 | 关键意图未表达或任务中断 |

#### 判定依据
| 信号 | 用途 |
|---|---|
| 目标槽位是否完整 | 如饮品、冷热、数量 |
| 对话轮次 | 是否推进到结单/确认 |
| NPC 是否理解 | 是否产生正确回应 |
| 用户是否依赖完整答案 | 判断掌握程度 |

---

### 模块 F：复盘页

#### 目标
对用户最重要的学习时刻：刚犯错、刚成功、刚记得住。

#### 功能需求
| 编号 | 功能 | 说明 |
|---|---|---|
| F1 | 任务结果 | 成功/部分成功/失败 |
| F2 | 高价值错误提示 | 只展示 1-3 个 |
| F3 | 更自然表达 | 给可替换版本 |
| F4 | 结构复看 | 查看 gloss |
| F5 | 跟说重练 | 再说一遍 |
| F6 | 迁移练习 | 换场景再用一次 |
| F7 | 加入表达库 | 收藏到“我的表达” |

#### 错误分类
| 类别 | 是否优先反馈 |
|---|---|
| 影响理解 | 高 |
| 影响任务完成 | 高 |
| 不自然但可懂 | 中 |
| 细微语法问题 | 低 |
| 发音细节 | 低（V1） |

---

### 模块 G：表达库（Personal Expression Bank）

#### 核心思想
不做“词汇本”，做“我真正会说的话”。

#### 功能需求
| 编号 | 功能 | 说明 |
|---|---|---|
| G1 | 收藏表达 | 用户在任务后保存 |
| G2 | 按场景分类 | 餐厅/社交/工作 |
| G3 | 变体显示 | 同一表达不同说法 |
| G4 | 结构直译显示 | 方便复习 |
| G5 | 快速复练 | 1 分钟复说 |

---

### 模块 H：学习画像

#### 目标
系统知道用户“卡在哪里”。

| 维度 | 示例 |
|---|---|
| 场景 | 餐厅强，打车弱 |
| 意图 | 能请求、不会澄清 |
| 表达模式 | 会“我要…”，不会“可以…吗” |
| 常错点 | 量词、声调、否定位置 |
| 英文依赖程度 | 对 gloss 依赖高/中/低 |

# 11. 交互与反馈策略

## 11.1 纠错原则

| 情况 | 系统怎么做 | 原因 |
|---|---|---|
| 表达可懂 | 不打断 | 保护流畅感 |
| 表达不自然但可懂 | 结束后反馈 | 不影响任务 |
| 表达导致误解 | NPC 用真实澄清方式追问 | 模拟现实 |
| 用户卡住 | 给逐步提示 | 保持挑战但不让其退出 |
| 连续失败 | 切换半引导模式 | 降低挫败感 |

---

## 11.2 提示层级设计

| 层级 | 内容 | 适用人群 |
|---|---|---|
| L0 | 无提示 | 已有基础用户 |
| L1 | 关键词提示 | A1 |
| L2 | 结构提示（Gloss） | A0-A1 |
| L3 | 半句提示 | 容易卡顿用户 |
| L4 | 完整参考句 | 紧急救援 |

# 12. 技术方案（优化版）

## 12.1 先说结论：别做“纯 AI 自由聊天”
最优方案不是“用户说什么 AI 都接”，而是：

> **任务状态机 + AI 对话编排器 + 结构直译引擎 + 复盘引擎**

这样才可控、可评估、可优化。

---

| 层级 | 模块 | 作用 |
|---|---|---|
| 客户端 | iOS / Android / Web | 任务浏览、对话、展示直译、语音交互 |
| API 层 | Gateway + Auth | 鉴权、路由、限流 |

| 数据层 | PostgreSQL + Cache | 用户画像、任务记录、表达库 |
| 内容层 | CMS/内容后台 | 编辑任务、审校 gloss、管理脚本 |

## 12.4 技术方案优化建议

### 优化 1：首版做“文字优先 + 语音增强”，不要“纯语音优先”
| 方案 | 结论 | 原因 |
|---|---|---|
| 纯语音优先 | 不建议首版 | 初学者中文发音 + ASR 容错难 |
| 文本优先 | 可行 | 容易稳定 MVP |
| 文本 + 语音增强 | **推荐** | 用户可说也可打，失败率最低 |

---

### 优化 2：Gloss 引擎不要完全依赖大模型
建议采用混合方案：

| 层 | 技术 |
|---|---|
| 基础层 | 分词、词性、词块规则 |
| 对齐层 | bilingual lexicon + pattern templates |
| 生成层 | LLM 生成结构直译与说明 |
| 质检层 | 规则校验 + 人工审核高频内容 |

#### 原因
完全由 LLM 做 literal gloss，质量会飘、风格会不一致。

---

### 优化 3：对话引擎必须“受控”
| 做法 | 原因 |
|---|---|
| 用任务状态机限制 LLM 回复边界 | 防止跑题 |
| 用 slot filling 判定是否完成任务 | 防止“聊得热闹但没学到” |
| 用 JSON 结构化输出 | 方便前端和评估 |

---

### 优化 4：反馈引擎分层
| 层级 | 内容 |
|---|---|
| 沟通成功层 | 你有没有把事情说成 |
| 表达自然层 | 有没有更自然的说法 |
| 结构理解层 | 这句为什么这么排 |
| 发音层 | 哪个音最值得修 |

> 不要把所有反馈混成一锅。那会让用户崩。

## 12.5 核心服务设计

### A. Scenario Engine（场景引擎）

| 子模块 | 功能 |
|---|---|
| Scenario Template | 场景模板，如餐厅、打车 |
| Mission State Machine | 任务状态流转 |
| NPC Persona | 角色说话风格 |
| Required Slots | 必须表达的信息点 |
| Branch Rules | 出错后的 NPC 合理反应 |
| Difficulty Layer | 新手/进阶模式 |

#### 示例：点咖啡任务状态机
```text
[开始]
  ↓
[问候]
  ↓
[表达饮品需求]
  ↓
[补充冷热/大小/糖冰]
  ↓
[确认订单]
  ↓
[付款]
  ↓
[任务完成]
```

---

### B. Gloss Engine（结构直译引擎）

#### 输出字段设计
| 字段 | 说明 | 示例 |
|---|---|---|
| hanzi | 中文原句 | 你叫什么名字 |
| pinyin | 拼音 | nǐ jiào shénme míngzi |
| chunks | 词块 | 你 / 叫 / 什么名字 |
| gloss | 结构直译 | you / are-called / what name |
| natural_en | 自然翻译 | what is your name |
| note | 极简语法备注 | “叫”表示名字/称呼 |
| level | 难度 | A0 |
| scenario | 所属场景 | social_intro |

#### 技术路线
| 步骤 | 说明 |
|---|---|
| 1 | 中文分词 |
| 2 | 词块切分 |
| 3 | 调用模式模板 |
| 4 | 词级英文映射 |
| 5 | LLM 调整成一致风格 gloss |
| 6 | 规则校验 |
| 7 | 高频内容人工审校 |

---

### C. Dialogue Orchestrator（对话编排器）

| 功能 | 说明 |
|---|---|
| 管理上下文 | 当前任务、当前 NPC、已完成槽位 |
| 调用 AI | 生成回复、判断理解 |
| 控制提示 | 何时给提示、给哪一层 |
| 控制纠错 | 即时 or 延迟 |
| 生成复盘数据 | 供 debrief 页面使用 |

#### 推荐输出结构（JSON）
```json
{
  "npc_reply_cn": "好的，您要热的还是冰的？",
  "npc_reply_en": "Okay, hot or iced?",
  "task_status": "in_progress",
  "slots_filled": ["drink_type"],
  "missing_slots": ["temperature"],
  "hint_level_recommended": 1,
  "user_message_understood": true,
  "error_tags": ["classifier_missing"]
}
```

---

### D. Debrief Engine（复盘引擎）

| 功能 | 说明 |
|---|---|
| 错误提炼 | 只选 1-3 个关键点 |
| 对话摘要 | 你刚刚完成了什么 |
| 表达替换 | 给更自然说法 |
| 迁移练习 | 同一表达换场景 |
| 加入表达库 | 用户沉淀成果 |

# 15. 内容生产机制

## 15.1 为什么不能全靠 AI 自动生成

- gloss 是否稳定

# 19. 关键风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| ASR 识别不准 | 初学者发音不稳 | 文本输入兜底；关键场景支持关键词识别 |
| Gloss 风格不稳定 | LLM 输出会飘 | 模板化 + 规则校验 + 人工审校 |
| AI 对话太散 | 聊得多但学不到 | 状态机约束 + 任务槽位判定 |
| 纠错过多 | 用户压力大 | 只反馈最关键 1-3 点 |
| 内容太假 | 像教材对话 | 用真实生活意图写任务 |
| 过早做“大而全” | 开发失控 | 先聚焦 20 个高频任务 |

# 20. 本方案相对原始想法的优化点

| 原始想法 | 优化后的方案 |
|---|---|
| 真实语言场景 | 变成“任务地图 + 状态机 + NPC” |
| 英文直译 | 升级为“三层显示模型” |
| 沉浸式学习 | 变成“场景热身 → 对话 → 复盘 → 迁移” |
| 在出错中学习 | 变成“延迟反馈 + 高价值纠错” |
| 不在乎翻译优雅 | 变成“结构直译优先，自然翻译保底” |
| 模仿现实学习 | 变成“沟通优先、解释随后、反复迁移” |

| 是否即时纠错 | 全程 / 关键时刻 / 结束后 | **结束后为主，关键错误即时** |
| 是否允许看完整答案 | 不允许 / 可按需 | **允许，但记为“支架依赖”** |
| 是否对齐 HSK | 强绑定 / 弱映射 / 不对齐 | **弱映射** |

## 21.3 技术实现
| 问题 | 选项 | 我的建议 |
|---|---|---|
| Gloss 生成 | 纯 LLM / 混合式 | **混合式** |
| 内容生产 | 全自动 / 专家+AI | **专家+AI** |
| 首版任务数 | 10 / 20 / 50 | **20** |
| 首版是否上发音评分 | 是 / 否 | **否，先不上重评分** |

如果只抓三件事，就是：

1. **高质量真实任务设计**
2. **结构直译引擎**
3. **任务后复盘与迁移**

## 22.2 不是所有 AI 学语言产品都值得做
但你这个方向值得做，因为它避开了最拥挤、也最虚假的路径：

- 不做“伪学习”
- 不做“假进步”
- 不做“背词幻觉”

## 0. 执行摘要

### 0.1 先说结论
这类产品的技术核心，不是“接个大模型做聊天”。

真正的技术壁垒是 4 件事：

| 核心能力 | 为什么重要 |
|---|---|
| **任务状态机** | 保证用户是在“完成任务”，不是闲聊 |
| **结构直译引擎（Gloss Engine）** | 帮用户理解中文结构，而不只是看自然翻译 |
| **容错对话编排器（Dialogue Orchestrator）** | 允许用户边错边说，并动态给支架 |
| **复盘与迁移引擎（Debrief Engine）** | 把一次对话变成长期可迁移能力 |

### 0.2 技术路线建议
**Straight shooting：MVP 不建议做成微服务大平台。**

建议路线：

- **架构**：模块化单体（Modular Monolith）优先
- **交互**：文本优先 + 语音增强
- **对话**：状态机约束 + LLM 生成，不做纯自由聊天
- **Gloss**：规则 + 模板 + LLM + 人工审校的混合方案
- **内容生产**：专家定义任务，AI 扩写，不做全自动内容工厂
- **埋点与评估**：从第一天就做，不然后面无法判断“是否真的有效”

# 1. 背景 → 现状 → 技术判断

## 1.1 背景
你要做的不是“学中文 App”，而是“**中文任务模拟器**”。

这和 [ACTFL Proficiency Guidelines](https://www.actfl.org/educator-resources/actfl-proficiency-guidelines/) 及 [CEFR](https://www.coe.int/en/web/common-european-framework-reference-languages) 强调的 can-do 能力是高度一致的：  
语言能力不等于知道知识，而是**能否在真实语境中完成任务**。

---

## 1.2 当前技术方案常见误区

| 误区 | 常见做法 | 结果 |
|---|---|---|
| 把 AI 当聊天机器人 | 用户随便聊，系统随便回 | 聊得热闹，但学不到关键能力 |
| 把学习单位做成词卡 | 词汇、例句、选择题 | 迁移弱，现实中不会说 |
| 纯自动生成内容 | 所有句子都让 LLM 临时生成 | 风格飘、gloss 不稳定、教学性差 |
| 过度做语音评分 | 一上来就细扣发音 | 初学者挫败感强 |
| 没有任务判定 | 没有“成功/失败/部分成功” | 无法衡量真实学习效果 |

---

## 1.3 技术判断
本产品最适合的不是“内容平台架构”，而是“**实时任务引擎架构**”。

## 3.2 架构风格建议

# 5. 核心模块设计

## 5.1 Mission Service（任务服务）

### 职责
管理：

- 场景（Scenario）
- 任务（Mission）
- 难度（Level）
- 状态机（State Machine）
- 必须完成的信息槽位（Required Slots）

### 数据结构示意

| 字段 | 说明 | 示例 |
|---|---|---|
| scenario_id | 场景ID | cafe_order |
| mission_id | 任务ID | order_hot_latte |
| level | 难度 | A0 |
| objective | 任务目标 | 点一杯热拿铁，不要冰 |
| required_slots | 必填槽位 | drink, temp |
| optional_slots | 选填槽位 | sugar, size |
| success_rules | 成功规则 | drink+temp 被理解 |
| allowed_branches | 允许分支 | clarify, confirm, payment |

### 状态机示意

```text
[Start]
  ↓
[Greet]
  ↓
[Request Drink]
  ↓
[Clarify Details]
  ↓
[Confirm Order]
  ↓
[Payment]
  ↓
[Complete]
```

### 为什么必须用状态机
因为这能解决一个核心问题：

> AI 聊天会很自然，但“教学任务”会失控。  
> 状态机不是为了限制自然度，是为了保证“学习目标达成”。

---

## 5.2 Dialogue Orchestrator（对话编排器）

### 职责
这是系统大脑。

它负责：

| 功能 | 说明 |
|---|---|
| 读取任务状态 | 当前在哪一步 |
| 读取用户画像 | 水平、常错点、支架依赖 |
| 解析用户输入 | 中文/拼音/混合/英文求助 |
| 调度模型 | 该调用哪个模型、哪个 prompt |
| 决定提示级别 | 不提示 / 结构提示 / 半句 / 完整参考 |
| 判断任务进展 | 已完成哪些槽位 |
| 记录错误事件 | 供复盘使用 |

### 单轮处理流程

```text
[用户输入]
   ↓
[Input Normalizer]
   ↓
[状态机检查]
   ↓
[槽位提取 / 理解判断]
   ↓
[是否需要提示 or 澄清]
   ↓
[生成 NPC 回复]
   ↓
[更新任务状态]
   ↓
[记录埋点与错误]
```

---

## 5.3 Input Normalizer（输入标准化）

这个模块很关键，很多产品忽略了。

### 你这个产品必须接受 4 种输入
| 输入类型 | 示例 | 是否支持 |
|---|---|---|
| 汉字 | 我要一杯热拿铁 | 是 |
| 拼音 | wo yao yi bei re na tie | 是 |
| 混合 | wo yao 一杯 latte | 是 |
| 英文求助 | how to say hot latte | 是，但标记为支架模式 |

### 处理逻辑
| 步骤 | 说明 |
|---|---|
| script detection | 检测是中文/拼音/混合/英文 |
| normalization | 去噪、标准化标点、大小写 |
| pinyin candidate mapping | 拼音转候选汉字 |
| rescue mode | 纯英文输入时，进入求助/支架模式 |
| confidence score | 标记识别置信度 |

### 优化建议
不要把纯英文输入当错误。  
对初学者来说，它是“**求生信号**”。

---

## 5.4 Gloss Engine（结构直译引擎）

### 核心目标
把中文句子拆成用户能理解的“结构化桥梁”。

### 设计原则
参考语言学中的 interlinear gloss 思路，但 UI 层不应做成学术化标注；可以借鉴 [Leipzig Glossing Rules](https://www.eva.mpg.de/lingua/resources/glossing-rules.php) 的一致性原则，但输出要更适合初学者。

### 输出三层结构

| 层 | 内容 | 示例 |
|---|---|---|
| L1 | 中文原句 | 你叫什么名字 |
| L2 | 结构直译 | you / be-called / what / name |
| L3 | 自然翻译 | what is your name |

### Pipeline 设计

```text
[中文句子]
   ↓
[分词]
   ↓
[教学词块切分]
   ↓
[结构模式匹配]
   ↓
[词块级英文映射]
   ↓
[LLM 调整风格]
   ↓
[规则校验]
   ↓
[缓存/人工审核]
```

### 为什么不用“纯逐字对齐”
因为中文不是所有句子都适合逐字硬绑。  
推荐采用：

> **词级 + 词块级 + 结构级混合对齐**

### Gloss 结果对象

```json
{
  "hanzi": "你叫什么名字",
  "pinyin": "nǐ jiào shénme míngzi",
  "chunks": ["你", "叫", "什么名字"],
  "gloss": ["you", "be-called", "what name"],
  "natural_en": "what is your name",
  "note": "叫 here means 'to be called / named'",
  "confidence": 0.93,
  "verified": true
}
```

### Gloss 质量校验规则

| 规则 | 目的 |
|---|---|
| 保留中文语序 | 帮助用户建立中文结构感 |
| 不做英语润色 | 避免过度自然化 |
| 相同模式输出一致 | 避免用户困惑 |
| 高频句优先人工审校 | 保证教学稳定性 |
| note 保持极简 | 避免又变成语法课 |

---

## 5.5 Dialogue Generation（NPC 回复生成）

### 目标
让 NPC 像真实人，但又不偏离任务。

### 推荐策略：双层生成
| 层 | 作用 |
|---|---|
| Policy Layer | 先决定下一步“行为意图” |
| Response Layer | 再生成自然中文回复 |

### Policy Layer 输出示例

```json
{
  "next_act": "ask_clarification",
  "reason": "temperature_missing",
  "task_status": "in_progress",
  "slots_filled": ["drink_type"],
  "missing_slots": ["temperature"],
  "should_hint_user": false
}
```

### Response Layer 输出示例

```json
{
  "npc_reply_cn": "好的，您要热的还是冰的？",
  "npc_reply_en": "Okay, hot or iced?",
  "emotion": "neutral_friendly"
}
```

### 为什么这样拆
因为如果直接让模型“一步生成一切”，问题会是：

- 输出结构不稳定
- 难评估
- 难 debug
- 难控制教学策略

---

## 5.6 Hint Engine（提示引擎）

### 核心原则
**不给太早，不给太满，不给太多。**

### 提示分级

| 级别 | 内容 | 示例 |
|---|---|---|
| H0 | 无提示 | 用户自由说 |
| H1 | 关键词提示 | 热 / 一杯 / 拿铁 |
| H2 | 结构提示 | I / want / one cup / hot latte |
| H3 | 半句提示 | 我要一杯 ___ 拿铁 |
| H4 | 完整参考 | 我要一杯热拿铁 |

### 升级规则建议

| 条件 | 提示升级 |
|---|---|
| 连续 1 次失败 | H1 |
| 连续 2 次失败 | H2 |
| 连续 3 次失败 | H3 |
| 明确点击“我卡住了” | 直接 H3/H4 |
| 高焦虑用户或 A0 | 起始可 H1 |

---

## 5.7 Debrief Engine（复盘引擎）

### 目标
把一次任务结束后的 30 秒，变成学习最有价值的时刻。

### 输入
| 输入项 | 说明 |
|---|---|
| 对话完整转录 | 用户和 NPC 每轮内容 |
| 错误事件 | 高频错、影响理解错 |
| 提示使用记录 | 依赖程度 |
| 任务状态 | 成功/失败/部分成功 |
| 目标表达 | 本任务核心 pattern |

### 输出
| 输出项 | 说明 |
|---|---|
| 做对了什么 | 先给正反馈 |
| 最值得改的 1-3 点 | 不贪多 |
| 更自然表达 | 迁移价值高 |
| 结构复看 | 再看 gloss |
| 下一步任务 | 同场景或近迁移 |

### 输出示例

```json
{
  "result": "success",
  "strengths": [
    "你成功表达了想要的饮品",
    "店员能够理解你的主要需求"
  ],
  "top_fixes": [
    {
      "type": "measure_word",
      "before": "我要一个热拿铁",
      "after": "我要一杯热拿铁",
      "reason": "饮品更自然用“一杯”"
    }
  ],
  "transfer_patterns": [
    "我要一杯冰美式",
    "我要一杯热奶茶"
  ],
  "next_mission_id": "bubble_tea_order_basic"
}
```

---

## 5.8 User Skill Service（用户能力画像）

### 不建议用“背过多少词”来建模
应该记录的是：

| 维度 | 示例 |
|---|---|
| 场景能力 | 餐厅强、打车弱 |
| 意图能力 | 会请求，不会澄清 |
| 结构能力 | 会“我要…”，不会“可以…吗” |
| 支架依赖 | 对完整参考依赖高 |
| 容错能力 | 出错后能否自行修复 |
| 英文依赖 | gloss 展开频率高/低 |

### 推荐计算字段

| 字段 | 说明 |
|---|---|
| success_rate_by_scene | 各场景任务成功率 |
| support_ratio | 完整提示使用比例 |
| repair_attempt_rate | 出错后再次尝试比例 |
| transfer_success_rate | 表达迁移成功率 |
| last_seen_patterns | 最近练过的表达模式 |

### 推荐算法：下一个任务排序
```text
Priority
= 真实需求权重
× (1 - 成功率)
× 高频度
× 迁移价值
× 时间衰减
```

# 8. Prompt 与模型编排设计

## 8.1 模型分工建议
别让一个模型做所有事。

| 任务 | 推荐模型类型 | 原因 |
|---|---|---|
| 意图/状态分类 | 小模型 / 规则 | 便宜、稳定、快 |
| NPC 回复生成 | 主力对话模型 | 需要自然度 |
| Debrief 复盘 | 主力或中型模型 | 要总结和教学性 |
| Gloss 生成 | 离线模型 + 规则 | 需要一致性 |
| ASR/TTS | 专项语音模型 | 专业工具更合适 |

---

## 8.2 Prompt 架构建议

### Prompt 1：Policy Prompt
目标：不直接生成话术，先判定“下一步该做什么”。

**输入：**
- mission objective
- current state
- required slots
- user input normalized
- skill level
- support history

**输出：**
- next_act
- updated slots
- task_status
- hint recommendation
- error tags

---

### Prompt 2：NPC Prompt
目标：根据 policy 输出具体话术。

**规则：**
- 必须扮演角色，不讲课
- 中文优先
- 不主动长篇解释语法
- 在用户可懂范围内自然
- 保持任务上下文一致

---

### Prompt 3：Debrief Prompt
目标：把对话转成可执行反馈。

**规则：**
- 先夸一条真实做对的
- 最多讲 3 个高价值问题
- 不讲长篇语法
- 每条反馈必须附一个可替换表达
- 输出固定 JSON

---

## 8.3 Structured Output 强约束
建议所有模型输出都走 JSON Schema / OpenAPI 风格约束，[OpenAPI](https://spec.openapis.org/oas/latest.html)

### 示例 Schema：Policy Output

```json
{
  "type": "object",
  "properties": {
    "next_act": { "type": "string" },
    "task_status": { "type": "string" },
    "slots_filled": { "type": "array", "items": { "type": "string" } },
    "missing_slots": { "type": "array", "items": { "type": "string" } },
    "hint_level_recommended": { "type": "integer" },
    "error_tags": { "type": "array", "items": { "type": "string" } }
  },
  "required": [
    "next_act",
    "task_status",
    "slots_filled",
    "missing_slots",
    "hint_level_recommended",
    "error_tags"
  ]
}
```

# 10. 评估与判定逻辑

## 10.1 任务成功判定

### 推荐评分公式
```text
Task Score
= 0.50 × 必需槽位完成度
+ 0.20 × 可理解度
+ 0.20 × 支架独立性
+ 0.10 × 自我修复能力
```

### 阈值建议

| 分数 | 判定 |
|---:|---|
| ≥ 0.75 | 成功 |
| 0.45 - 0.74 | 部分成功 |
| < 0.45 | 失败 |

---

## 10.2 错误优先级逻辑

| 类型 | 例子 | 优先级 |
|---|---|---|
| 影响理解 | 热/冰说反、地点说错 | 高 |
| 影响任务完成 | 缺数量、缺对象 | 高 |
| 高迁移价值问题 | 量词、基本句型 | 高 |
| 不自然但可懂 | 一个/一杯 | 中 |
| 细节语法/语气 | 很细微的问题 | 低 |
| 发音小问题 | 不影响理解 | 低 |

### 规则
复盘只取 **Top 1-3**，不要全量倾倒。

---

## 10.3 Gloss 质量评估

| 指标 | 目标 |
|---|---|
| 结构一致性 | 同模式输出一致 |
| 语序保留率 | 高 |
| 可读性 | 初学者可理解 |
| 误导率 | 低 |
| 人工审校通过率 | 高频内容 > 95% |

# 17. 技术方案优化建议（重点）

## 17.1 方案优化 1：高频任务模板化，长尾任务生成化
| 类型 | 处理方式 |
|---|---|
| 首批 20 个核心任务 | 模板 + 状态机 + 审校 gloss |
| 任务变体 | LLM 扩写 |
| 长尾开放任务 | 后续再加 |

### 价值
这是质量和效率的最佳平衡。

---

## 17.2 方案优化 2：Gloss 前置生产，不要全实时
| 策略 | 原因 |
|---|---|
| 高频表达预生成 | 保证一致性 |
| 用户临时句子按需生成 | 节省成本 |
| 复盘建议里常用句缓存 | 降低延迟 |

---

## 17.3 方案优化 3：先“任务成功”，后“语言优美”
| 优先级 | 解释 |
|---|---|
| 1 | 是否被理解 |
| 2 | 是否完成任务 |
| 3 | 是否更自然 |
| 4 | 是否更地道 |

这是产品哲学，也是系统评分逻辑。

---

## 17.4 方案优化 4：用状态机约束，保留 AI 的自然度
不要在“规则系统”和“生成系统”之间二选一。  
要做的是：

> **规则负责边界，AI 负责自然度。**

## 21.5 核心评分公式（已实现）

```text
Task Score (0-1)
= (comprehensionRate × 60 + turnEfficiency × 40 - errorPenalty - hintPenalty) / 100

其中：
- comprehensionRate = understoodTurns / totalTurns
- turnEfficiency = max(0, 1 - (totalTurns - requiredSlots.length) × 0.1)
- errorPenalty = errorTurns.count × 5
- hintPenalty = hintsUsed × 3

判定阈值：
- score ≥ 0.6 → 成功（Success）
- score < 0.6 且有对话记录 → 部分成功（Partial Success）
- 无对话记录 → 失败（Failure）
```

## 21.6 已知限制与后续优化方向

| 无语音功能 | 仅文本输入 | 接入 ASR/TTS（Whisper + Edge TTS） |

## 22.1 AI API 集成

### 架构设计

| Prompt 层 | 职责 | 输入 | 输出 |
|---|---|---|---|

### API 路由

| 路由 | 方法 | 功能 |
|---|---|---|
| `/api/chat` | POST | 处理单轮对话（Policy→NPC 两步调用） |
| `/api/debrief` | POST | 生成任务复盘报告 |

### 降级策略
```text
用户请求 → API Route → AI 可用？
                         ├─ 是 → Policy Prompt → NPC Prompt → 返回
                         └─ 否 → 本地规则引擎 → 返回
```

- AI 不可用时自动降级到规则引擎，零感知
- 支持环境变量 `USE_RULE_ENGINE=true` 强制使用规则引擎
- API 调用失败时 catch 降级，保证服务可用性
